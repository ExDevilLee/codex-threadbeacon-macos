# 状态数据源健康诊断设计

## 背景

ThreadBeacon 依赖 Codex 本地 SQLite、session index、rollout JSONL 和结构化日志推导任务
状态。当前主数据源读取失败会显示一条错误，但 rename、rollout 或日志等可降级数据源失败时
通常静默回退。用户看到状态或标题不符合预期时，无法区分“真实任务状态”和“数据源降级”。

本功能的目标是让用户判断当前状态灯是否值得信任，而不是把 ThreadBeacon 扩展成日志或
数据库管理工具。

## 实现状态

2026-07-18 MVP 已按本文方案实现：刷新结果携带健康报告，底部提供紧凑入口，popover
按需展示四类数据源和最后成功刷新时间。健康状态不持久化、不触发提示音，也不改变现有
读取范围。当前等待 Lee 完成实机交互验收。

## 已确认方案

采用“刷新结果携带健康报告 + 底部紧凑入口 + 按需详情”的方案。

不采用以下方案：

- 独立定时探测：会重复读取文件，且探测结果可能与当前任务快照不属于同一次刷新。
- 只显示最后一次错误：无法区分核心数据源与可选增强数据源，也无法表达部分降级。
- 独立诊断窗口：首版信息量很小，会增加窗口和导航复杂度。

## 产品范围

MVP 包含四类数据源：

| 数据源 | 作用 | 失败后的产品状态 |
| --- | --- | --- |
| 任务数据库 | 任务列表、归档、Token、父子关系 | `不可用`，本次刷新失败 |
| Rename 索引 | Codex App rename 后标题 | `部分降级`，回退 SQLite 标题 |
| Rollout | 任务状态、活动时间、Token 明细 | `部分降级`，按失败任务数汇总 |
| 服务日志 | 400/429/503 异常与模型容量错误 | `部分降级`，不显示服务异常 |

MVP 不包含：

- 原始错误、完整路径、任务标题、任务 ID、日志正文或会话内容展示。
- 自动修复、重建文件、修改 Codex 数据或调用 Codex App 私有接口。
- 历史健康趋势、遥测、通知或导出诊断报告。
- 把 Subagent、归档恢复 CLI 或提示音资源拆成新的健康数据源。

## 状态语义

整体健康状态只有三种：

- `正常`：本次刷新使用的数据源全部成功；未使用的数据源不算失败。
- `部分降级`：任务列表仍可显示，但至少一个可选数据源失败或部分 rollout 无法读取。
- `不可用`：核心任务数据库失败，本次刷新无法生成可信任务列表。

每个数据源使用 `正常`、`部分降级`、`不可用`或`未使用`状态，并可带一条稳定、脱敏的
说明。rollout 只展示“成功读取 N 个／失败 N 个”，不显示对应任务。

`lastSuccessfulRefreshAt` 只在核心任务列表成功刷新后更新。核心失败时保留上一批任务快照，
同时把整体健康状态改为`不可用`，避免清空列表造成二次干扰。

## 架构与数据流

### `DataSourceHealthReport`

位于 `ThreadBeaconCore/Models`，保存整体状态、四个数据源状态、rollout 成功／失败数量和
最后一次成功刷新时间。模型负责派生整体状态和稳定的用户说明，UI 不自行推断健康语义。

### `ThreadStatusLoader`

新增返回 `ThreadStatusLoadResult` 的加载入口，一次刷新同时生成任务快照与健康报告：

1. 核心 SQLite 成功后标记任务数据库正常。
2. 捕获 rename、服务日志和每个 rollout 的失败并记录降级，不暴露原始错误。
3. 保留现有回退行为和返回 `[ThreadSnapshot]` 的兼容入口，减少无关调用方改动。
4. 核心 SQLite 失败时抛出带健康报告的结构化错误。

### `ThreadStatusStore`

发布当前 `dataSourceHealth`。成功刷新时同时更新任务快照、健康报告和最后成功时间；核心失败
时发布不可用报告但保留既有快照。通知仍只根据成功生成的任务快照计算，健康状态变化不播放
提示音。

### SwiftUI

底部右侧放置一个紧凑健康图标：正常使用中性色，部分降级使用黄色，不可用使用红色。悬浮
显示一句整体说明，点击打开 popover，列出最后成功刷新时间和四个数据源状态。

入口不占用任务行，不新增默认列。详情使用系统图标、文字和颜色共同表达，不能只依赖颜色。

## 错误与隐私边界

- UI 只展示预定义的稳定错误类别和数量，不直接展示 `localizedDescription` 原文。
- 不保存健康历史；报告只存在当前进程内存中。
- 不增加文件读取范围、网络请求、遥测、Accessibility 权限或 Codex 数据写入。
- 数据源未参与本次刷新时显示`未使用`，不误报为正常或失败。

## 测试与验收

自动测试至少覆盖：

- 所有已使用数据源成功时整体为正常。
- rename 或日志失败时任务仍返回，整体为部分降级。
- 部分 rollout 失败时计数准确并继续使用回退状态。
- 核心 SQLite 失败时产生不可用健康报告。
- Store 成功刷新更新时间和健康报告；失败时保留旧快照并发布不可用报告。

最终验收：

- 正常数据下 App 构建启动，底部健康入口显示正常。
- 点击入口能看到四类数据源及最后成功刷新时间。
- UI 和公开文档不包含真实本机路径、任务标题、任务 ID 或原始日志。
- 全量测试、Markdown lint 和 `git diff --check` 通过。
