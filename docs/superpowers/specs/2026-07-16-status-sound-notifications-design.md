# 多状态提示音设计

## 目标

让用户不持续注视 ThreadBeacon 窗口，也能区分 Codex 主任务的关键结果和待处理事件，
同时避免自动重试、历史状态回放和多个任务集中变化造成声音噪音。

提示音服务于“是否需要用户关注”，不把每一种底层错误或 HTTP 状态码映射为独立
声音。本设计先建立统一事件模型，再按数据源的真实可靠程度分阶段启用。

## 已确认的产品决策

- 可恢复错误正在自动重试时保持静音。
- 只有重试耗尽、任务停止或需要人工介入时才播放警告或失败提示音。
- 默认只为主任务播放提示音，不聚合或单独通知 subagent。
- 用户主动中断任务默认静音。
- App 启动、恢复监听和普通刷新不能补播历史事件。
- 第一阶段只启用已有可靠事件证据支持的提示类型；不能通过宽泛文本搜索伪造状态。

## 事件分类与声音语义

| 类别 | 触发语义 | 默认声音特征 | 默认开关 |
| --- | --- | --- | --- |
| `done` | 当前 turn 成功完成 | 短促、柔和、明确的成功音 | 开启 |
| `attention` | 等待命令、文件修改、额外权限或用户输入 | 辨识度高的双音提示 | 开启 |
| `warning` | 可恢复异常的重试已经耗尽，需要关注 | 克制的警告音 | 开启 |
| `failure` | 不可恢复错误导致当前 turn 失败 | 比警告更严肃的失败音 | 开启 |
| `interrupted` | 用户取消或主动中断 | 不播放 | 关闭 |

`429`、`503`、服务过载和连接中断不是独立声音类别。它们需要结合结构化错误类型、
`willRetry` 和最终 turn 状态归入 `warning` 或 `failure`。同一个错误仍在自动恢复时不应
打扰用户。

## 数据源证据与边界

### Rollout

当前本地样本中，`event_msg.payload.type == task_complete` 是稳定的完成信号，可用于
识别 `done`。`turn_aborted` 样本目前只验证到用户中断语义，不足以推导警告或失败。

现有 rollout 样本没有验证到可稳定覆盖以下事件的专用结构：

- `429`、`503` 或其他终止错误及其重试结果。
- 命令执行、文件修改和额外权限审批。
- 所有需要用户输入的交互。

因此，第一阶段不能依赖错误字符串或对话正文扫描来启用 `attention`、`warning` 和
`failure`。这也保持当前“不读取会话正文和摘要”的隐私边界。

### Codex app-server

当前安装的 Codex CLI 实验性 app-server 协议提供更完整的结构化信号：

- `turn/completed` 和 turn 的 `completed`、`interrupted`、`failed`、`inProgress` 状态。
- 带 `willRetry`、结构化错误类型和可选 HTTP 状态码的 error notification。
- 命令执行、文件修改和额外权限的 approval request。
- 实验性的用户输入 request。

但 ThreadBeacon 单独启动的 app-server 进程尚未证明能够订阅 Codex Desktop 当前
app-server 实例中的事件；本机也没有可直接连接的受管 daemon/control socket。该路径
必须先做隔离 POC，不能按协议文件存在就视为集成已经可用。

## 分阶段交付

### 阶段一：可靠完成提示

- 建立统一 `SoundNotificationEvent` 模型和声音播放服务。
- 使用 rollout 的新 `task_complete` 转换识别 `done`。
- 实现启动基线、去重、暂停/恢复语义和声音总开关。
- 提供少量 App 内置短音效，并支持完成音试听和选择。
- 其他事件类别保留在模型与测试契约中，但不在 UI 中假装可配置或可用。

### 阶段二：app-server 集成 POC

隔离验证以下问题：

1. 独立 App 能否获得与 Codex Desktop 主任务一致的 `threadId` 和 `turnId`。
2. 能否实时收到完成、错误、重试和 approval request，而不是只看到自身创建的会话。
3. app-server 生命周期、重连、Codex 升级和协议变化如何处理。
4. 只读监听是否会引入控制权限、额外进程或明显资源消耗。
5. 事件是否仍能保持主任务过滤，并避免读取或持久化对话正文。

只有 POC 能稳定关联真实 Codex Desktop 任务时，才进入阶段三。

2026-07-17 的 POC 结果为受阻：独立 app-server 能从共享持久化数据列出 Desktop 任务
ID，但状态均为 `notLoaded`，无法看到 Desktop 已加载线程或接收其实时事件。现阶段不
进入阶段三，完整证据见
[`docs/app-server-integration-poc.md`](../../app-server-integration-poc.md)。

### 阶段三：待操作、警告与失败

- approval request 和可靠的用户输入 request 映射为 `attention`。
- `willRetry == true` 的中间错误保持静音。
- 重试耗尽但属于临时服务或连接问题时映射为 `warning`。
- 非重试错误和最终 `failed` 状态映射为 `failure`。
- Settings 开放四类事件的独立开关、声音选择和试听。

如果 app-server POC 不成立，阶段一仍作为完整、可信的最小功能保留；后续等待 Codex
提供共享事件接口，不退回到正文或错误文本猜测。

## 事件状态与去重

每次刷新产生任务快照，但声音只能由“本次观察到的新事件”触发，不能仅因当前状态
等于完成就播放。

事件身份使用：

```text
threadId + turnId + eventCategory
```

若 rollout 暂时没有独立 `turnId`，阶段一使用已验证的 turn 边界和完成事件位置生成
稳定的本地事件身份，不能只用线程 ID。播放后将身份写入有界的本地去重记录，避免：

- 两秒轮询重复播放同一完成事件。
- App 重启后再次播放最近的旧完成事件。
- rollout 文件尾部在多次读取时重复触发。

首次启动先建立当前事件水位，不播放已存在事件。去重记录只保存事件身份、类别和必要
时间戳，不保存标题、消息正文、reasoning 或工具参数，并定期清理过期项。

## 暂停、恢复与并发规则

- 暂停监听期间不读取自动刷新事件，也不播放提示音。
- 暂停期间的手动刷新只更新界面，默认不播放提示音。
- 恢复监听时先把当前事件作为新水位，再继续监听，不补播暂停期间事件。
- App 启动和数据源重连采用相同的基线规则。
- 同一刷新批次出现多个事件时，按优先级选择：
  `attention > failure > warning > done`。
- 同一优先级多个任务集中变化时只播放一次，并在界面保留各任务自身状态。
- 已经开始播放的短音效不被中途打断；同批次低优先级声音直接合并。

## Settings 设计

阶段一提供最小声音设置：

- 提示音总开关。
- 完成提示音开关。
- 内置声音选择。
- 试听按钮。

阶段三再按事件类别提供独立开关、声音选择和试听。音量默认跟随 macOS 系统输出，
第一版不增加 App 内独立音量滑块。

首版使用项目自行制作或许可清晰的 App 内置短音效，不能复制 prior-art 项目的音效包。
用户自定义文件属于后续候选，需要另行处理格式、Sandbox 文件权限、书签持久化和文件
丢失回退。

## 错误与降级策略

- 音频设备不可用或播放失败：不影响状态刷新和 UI，记录可诊断但不含隐私的信息。
- 用户选择的内置音效缺失：回退到默认声音，不重复弹窗。
- app-server 断开：禁用依赖它的事件识别并回退到 rollout 的可靠能力，不猜测状态。
- 协议出现未知错误类型：只有最终 `failed` 已确认时才播放通用失败音，否则保持静音并
  记录兼容性诊断。
- ThreadBeacon 自身读取失败不等于 Codex 任务失败，不能播放任务失败音。

## 测试与验收

### 阶段一自动测试

1. 从运行中观察到新的 `task_complete` 时只生成一次 `done` 事件。
2. 首次启动面对已完成任务不播放。
3. 两秒重复刷新、重启和 rollout 尾部重复数据不重复播放。
4. 暂停期间和恢复后的基线刷新不补播。
5. 手动刷新默认不触发声音。
6. subagent 完成不生成提示音。
7. 多个主任务同批完成只播放一次完成音。
8. 总开关或完成开关关闭时不播放，但状态刷新正常。
9. 试听不进入事件去重记录，也不改变任务状态。
10. 音频播放失败不影响 Store 和 UI。

### 阶段一真实使用验证

- 验证日期：2026-07-16。
- 已在真实 Codex 主任务完成后听到自动播放的 `done` 提示音，不是通过试听按钮触发。
- 设置入口和 Beacon、Chime、Pulse 三个内置声音的试听均已人工验证。
- 本次证据只验证阶段一完成音链路，不代表授权、警告或失败事件源已经可用。

### app-server POC 验收

1. 对真实 Codex Desktop 主任务完成一次可审计的 ID 关联。
2. 分别捕获成功完成、自动重试、最终失败和至少一种 approval request。
3. 证明自动重试期间静音，重试耗尽后只产生一个最终事件。
4. 验证断开、重启和 Codex Desktop 退出后的恢复行为。
5. 确认不持久化会话正文、命令内容、文件 diff 或用户输入内容。

### 阶段三自动测试

1. approval request 映射为 `attention` 并高于同批次 `done`。
2. `willRetry == true` 不产生声音事件。
3. 重试耗尽的临时错误映射为 `warning`。
4. 非重试终止错误映射为 `failure`。
5. 用户中断映射为 `interrupted`，默认不播放。
6. 每类开关、声音选择、试听和降级回退互不影响。

## 本设计不做

- 为每个 HTTP 状态码配置独立声音。
- 在提示音或系统通知中朗读任务标题或会话内容。
- 默认通知 subagent。
- 通过声音直接执行审批、重试或控制 Codex。
- 在 app-server POC 通过前承诺 `attention`、`warning` 和 `failure` 的交付时间。
- 第一版支持用户导入自定义声音或提供 App 内独立音量控制。
