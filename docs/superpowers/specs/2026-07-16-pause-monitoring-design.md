# 暂停与恢复监听设计

## 目标

允许用户临时暂停 ThreadBeacon 每两秒执行一次的自动刷新，在需要稳定查看详情、减少
本地读取或暂时不关心状态变化时保持当前界面不变，并能一键恢复监听。

## 已确认交互

- 标题栏在图钉和手动刷新之间增加监听控制按钮。
- 正在监听时显示 `pause.fill`，help 和无障碍标签为“暂停监听”。
- 已暂停时显示 `play.fill`，使用强调色，help 和无障碍标签为“恢复监听”。
- 点击暂停后停止后续每两秒自动刷新；如果一次只读刷新已经开始，允许该次完成。
- 点击恢复后立即刷新一次，然后继续每两秒自动刷新。
- 暂停期间手动刷新按钮保持可用；正在刷新时仍沿用现有 disabled 行为。
- 暂停状态只属于当前 App 进程，不写入 `UserDefaults`。重新启动 App 时默认恢复监听。

## 状态反馈

监听中沿用现有 Footer：

```text
更新于 18:00:50
```

暂停后显示：

```text
监听已暂停 · 上次更新 18:00:50
```

如果尚未完成第一次刷新就暂停，则显示“监听已暂停 · 尚未更新”。错误信息仍优先于
暂停提示，避免隐藏真实读取失败；错误存在时，标题栏按钮状态仍能表达当前是否暂停。

## 实现边界

`ContentView` 使用 view-local `@State` 保存 `isMonitoringPaused`，因为该状态：

- 不需要跨窗口或跨启动共享。
- 不属于任务数据，也不应进入 `ThreadStatusStore`。
- 不影响手动调用 `store.refresh()`。

自动刷新任务改为以暂停状态为 identity 的 SwiftUI `.task(id:)`：

- `false`：立即刷新，之后执行“休眠两秒 → 刷新”的循环。
- `true`：不进入循环。
- 状态切换时 SwiftUI 取消旧任务；恢复时创建新任务并立即刷新。

现有 `ThreadStatusStore.refresh()` 的并发保护保持不变。恢复或手动刷新遇到已有刷新时
会安全返回，不产生并发读取。

## 测试与验收

1. 保留全部 Core 单元测试，确认控制改动不影响状态和 Token 数据管线。
2. Debug 与 release build 均通过。
3. 真实 App 中确认暂停按钮停止自动更新时间变化至少五秒。
4. 暂停期间点击手动刷新，确认更新时间只前进一次，随后继续保持不变。
5. 点击恢复，确认立即刷新并继续按两秒节奏更新。
6. 暂停和恢复时 Footer、图标、help 与 accessibility label 同步变化。
7. 退出并重启 App，确认不会保留暂停状态。

## 本阶段不做

- 自定义刷新间隔。
- 持久化暂停状态。
- 菜单栏或快捷键控制。
- 按任务分别暂停；当前控制作用于整个 ThreadBeacon 窗口。
- 真正停止 Codex 任务；该功能只暂停 ThreadBeacon 的状态读取。
