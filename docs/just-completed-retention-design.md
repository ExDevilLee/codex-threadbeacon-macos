# “刚完成”状态保留时间设置设计

> 实现状态：已完成。配置模型、持久化、主任务与 Subagent 状态推导、Settings Picker、即时基线刷新
> 和中英文文案均已落地。

## 背景

ThreadBeacon 原先把任务完成后的 `justCompleted` 状态固定保留 60 秒，之后显示为 `idle`。
不同用户查看小屏幕的频率不同，固定一分钟可能来不及看到，因此需要提供有限、明确的可配置时长。

## 目标

- 在 Settings 的“任务监听”区域增加“刚完成保留时间”。
- 提供 `1、2、3、4、5 分钟` 五个选项，默认保持 `1 分钟`。
- 修改后立即重新计算当前任务状态，不需要重启 App。
- 主任务与 Subagent 使用同一个保留时间，保持状态语义一致。
- 只改变状态显示时长，不重放完成提示音，也不改变完成事件时间。

## 方案

采用与“刷新间隔”和“最大显示任务数”一致的 `Picker`，不提供自由输入。

固定选项能避免 `0`、负数和过大数值，也减少输入校验、单位歧义与设置页布局成本。当前需求范围只有
一分钟到五分钟，自由输入没有带来足够收益；如后续真实使用需要更长时长，再扩展选项即可。

## 配置模型

在 `DisplaySettings` 中增加：

- `supportedJustCompletedRetentionMinutes = [1, 2, 3, 4, 5]`
- `defaultJustCompletedRetentionMinutes = 1`
- `justCompletedRetentionMinutes`

`DisplaySettingsRepository` 使用独立 `UserDefaults` 键保存。旧版本没有该键时回退到默认一分钟；非法
持久化值也回退默认值，不影响其他显示设置。

## 数据流

1. Settings 中选择新的分钟数，并由 `@AppStorage` 持久化。
2. `ContentView` 监听该值变化，更新 `ThreadStatusStore` 的完成状态保留时长。
3. Store 立即执行一次基线刷新，避免把现有完成事件当作新的通知事件。
4. `ThreadStatusLoader` 使用传入的时长判断 `justCompleted` 是否已经转换为 `idle`。
5. 后续两秒刷新继续使用相同配置，不增加新的计时器。

## 边界行为

- 从较长时间改为较短时间：已经超过新阈值的任务在下一次刷新变为 `idle`。
- 从较短时间改为较长时间：仍可从 rollout 的完成时间重新推导为 `justCompleted`，但不补播提示音。
- 暂停监听时：配置会保存，恢复监听或手动刷新后生效。
- 缺少可靠完成时间时：沿用现有状态回退逻辑，不根据设置猜测。

## 测试与验收

- `DisplaySettings` 覆盖默认值、`1～5` 合法值及非法值回退。
- Repository 覆盖新字段持久化和旧配置迁移。
- Loader 覆盖一分钟与五分钟边界、超过阈值转为空闲，以及 Subagent 使用相同时长。
- Store / View 覆盖配置变化触发基线刷新，确认不会补发完成通知。
- 人工检查简体中文、英文、浅色与深色模式下的 Picker 布局。

## 非目标

- 不支持秒级或自由输入。
- 不改变 `running` 新鲜度、异常状态持续时间或任务排序优先级。
- 不增加每个任务单独配置的保留时间。
