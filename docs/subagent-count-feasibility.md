# Subagent 数量标记可行性调研

## 结论

Feature 5 可以进入最小设计，但第一版只能可靠显示“该主任务记录了多少个直接
Subagent”，不能把 SQLite 中的 `open` 数量解释为“当前正在运行的 Subagent”。

建议的最小范围：

- 主任务存在直接 Subagent 时，在任务行显示紧凑的总数标记。
- 数量来自 `thread_spawn_edges` 的直接父子关系，不读取子任务正文。
- 同时用 `child_thread_id` 排除子任务，补强现有仅依赖 `thread_source` 的过滤。
- 不显示活动数量，不按 `open` / `closed` 着色，不提供展开详情。

## 调研范围

本次在 2026-07-17 对本机 `~/.codex/state_5.sqlite` 做只读检查，并将关系表结果与
对应 rollout 尾部的 turn/final 事件做匿名聚合。没有输出任务 ID、标题、工作目录、
Subagent 名称或会话内容。

本次结论适用于当前 Codex 数据格式。SQLite schema 和字段语义不是稳定公开 API，
Codex 升级后仍需回归验证。

## 数据证据

### 父子关系覆盖完整

当前 `thread_spawn_edges` 包含：

| 指标 | 数量 |
| --- | ---: |
| 父子关系 | 121 |
| 不重复父任务 | 29 |
| 不重复子任务 | 121 |
| 找不到父任务记录的关系 | 0 |
| 找不到子任务记录的关系 | 0 |

每个有 Subagent 的父任务包含 1 至 23 个直接子任务，平均 4.17 个。当前 App 最近
8 个候选主任务中，有 2 个任务各自包含 3 个直接子任务，说明数量标记在当前真实列表
中已有可见样本。

### `open` 不等于正在运行

关系状态当前分布为：

| 关系状态 | 数量 |
| --- | ---: |
| `open` | 82 |
| `closed` | 39 |

但 82 条 `open` 关系全部至少 1 天没有更新，其中 71 条超过 7 天。将子任务 rollout
尾部按现有 ThreadBeacon 规则匿名归类后：

| 关系状态 | 已有 final | 仍有未闭合 turn |
| --- | ---: | ---: |
| `open` | 62 | 20 |
| `closed` | 29 | 10 |

这构成双向反例：`open` 中大量子任务已经有 final，`closed` 中也存在未闭合 turn。
因此 `open` / `closed` 更像关系生命周期或协作工具内部管理状态，不能作为实时运行灯。

### 现有子任务过滤存在兼容缺口

当前主列表通过 `thread_source <> 'subagent'` 排除子任务。但关系表中的 82 个历史
子任务没有 `thread_source` 值；其中 58 个仍未归档，会通过现有主任务筛选。它们目前
没有进入最近 8 条，所以当前界面未复现混入问题，但未来可能重新出现。

关系表中的 `child_thread_id` 是更直接的身份依据。实现数量标记时，应把主列表过滤
改为同时排除所有出现在 `thread_spawn_edges.child_thread_id` 中的任务。

## 推荐数据口径

第一版只建立以下口径：

```text
Subagent 总数 = 当前主任务在 thread_spawn_edges 中的直接 child_thread_id 数量
```

不采用以下口径：

- `open` 数量等于运行中数量。
- 子任务 rollout 未闭合就等于仍在运行。
- 把 Subagent 状态提升为主任务状态灯。
- 默认汇总 Subagent Token 到主任务 Token。

当前样本没有 Subagent 再创建下一层 Subagent 的记录。第一版只显示直接子任务数量，
如果后续出现多层关系，再单独决定显示直接数量还是整棵任务树数量。

## 最小产品设计建议

任务行仅在总数大于 0 时显示一个不抢占主状态灯注意力的紧凑标记，例如图标加数字。
悬浮说明使用“3 个 Subagent”这类明确文案。数字保持中性色，不使用红、黄、绿色，
避免暗示运行状态或异常程度。

第一版不加入点击展开。这能先回答真实产品问题：用户是否会利用 Subagent 数量理解
任务复杂度，以及这个信息是否值得占用小窗口空间。

## 实现边界

建议后续实现时：

1. SQLite 查询一次性返回主任务及直接 Subagent 总数，避免每行单独查询。
2. 主任务筛选同时使用 `thread_source` 和 `thread_spawn_edges.child_thread_id`。
3. `ThreadSnapshot` 只增加非负的 `subagentCount`，不暴露关系状态。
4. 数量为 0 时不渲染标记，保持当前列表密度。
5. 用临时 SQLite fixture 覆盖新格式、缺失 `thread_source` 的历史格式和孤立关系。

## 决策建议

建议进入最小 MVP 设计与开发，范围为：

- 显示直接 Subagent 总数。
- 修复关系表已知子任务的主列表过滤。
- 增加查询和 UI 测试。

活动数量、状态颜色、展开详情和 Token 聚合继续保留在 Roadmap，等待出现可靠实时数据
或总数标记经过真实使用验证后再决定。
