# 自动恢复设置与正式版界面设计

> 实现状态：已完成。Core 配置、终止异常候选、Accessibility 参数化发送、Release/Debug 界面隔离
> 和中英文文案均已落地；发送后恢复原前台 App 仍属于后续阶段。

## 背景

Accessibility POC 已验证以下主链路：按任务 ID 打开目标任务、使用 rename 标题二次确认、
检查输入框草稿、写入提示词、触发发送，并通过目标 rollout 中的新 `user_message` 与
`task_started` 确认结果。

当前 Settings 的“自动恢复”页仍直接展示只读诊断、输入框验证、目标任务 ID 和测试发送入口。
这些控件适合开发验证，不适合正式版本。与此同时，异常恢复提示词仍是代码中的固定文本，用户
无法按异常类型决定是否恢复或配置发送内容。

## 目标

- Release 构建不显示手工诊断、任务定位和测试发送控件。
- Debug 构建保留完整 POC 入口，便于本地复现 Codex Accessibility 兼容性问题。
- 正式设置页提供总开关，并允许按异常类型配置开关和提示词。
- HTTP 503 可配置但默认关闭，且仅在重试耗尽成为终止异常后触发。
- 未授权时不控制 Codex App，不回退到外部 `codex exec resume`。
- 保留现有异常自动恢复记录，用于观察候选异常、跳过、发送失败和发送成功。

## 非目标

- 本阶段不实现发送完成后恢复原前台 App。
- 本阶段不增加重试次数、冷却时间、规则脚本或自定义 HTTP 状态码。
- 不支持提示词模板变量，不把任务标题、任务 ID 或错误正文拼入消息。
- 不改变异常检测数据源和状态灯语义。

## 方案比较

### 方案 A：编译条件隔离开发工具（采用）

使用 `#if DEBUG` 只在 Debug 构建中挂载诊断区；Release 构建完全不创建这些控件。正式配置与
异常记录在两种构建中都可见。

优点是发布产物没有误触测试按钮的路径，且不需要额外维护开发者开关。缺点是排查用户机器上的
Accessibility 兼容问题时，需要提供 Debug 构建或让开发者本地复现。

### 方案 B：隐藏的开发者模式

通过连续点击版本号或隐藏开关显示诊断区。它便于现场排查，但测试发送能力仍存在于 Release
产物，增加误开启风险和产品解释成本，因此不采用。

### 方案 C：独立诊断 App 或 Target

把 POC 工具拆成独立产物，隔离最彻底，但当前代码量和使用频率不足以支撑额外 target、签名和
发布流程。本阶段不采用；若未来需要向用户收集 Accessibility 兼容证据再重新评估。

## 正式设置界面

“自动恢复”页由三部分组成：

1. **自动恢复总开关**
   - 默认关闭。
   - 开启时说明需要 macOS Accessibility 权限。
   - 未授权时显示“请求授权”和“打开辅助功能设置”；配置仍可编辑和保存。
   - 未授权或权限被撤销时不发送，只对新出现且规则已启用的异常写入“未发送”记录。
2. **异常规则**
   - 每种异常显示独立开关、提示词编辑框和“恢复默认”按钮。
   - 提示词去除首尾空白后不能为空，最大 500 个字符。
   - 无效提示词不保存，并在当前规则下方显示内联错误。
   - 第一版不做嵌套卡片，使用紧凑的 `Form` / `Section` 布局，避免 Settings 高度失控。
3. **自动恢复记录**
   - 保留现有记录列表和清空功能。
   - 记录实际采用的异常分类与提示词快照，后续修改配置不改写历史记录。

Debug 构建在页面底部增加“开发者诊断”折叠区，承载现有五类 POC 控件：只读访问、输入框验证、
任务 ID、目标切换验证和测试发送。Release 构建不渲染该区域，也不显示对应状态。

## 异常类型与默认值

| 类型 | 触发条件 | 类型默认值 | 默认提示词 |
| --- | --- | --- | --- |
| HTTP 400 | 终止型 `Bad Request` | 开启 | 刚才请求异常中断了，请继续未完成的任务 |
| HTTP 429 | 重试耗尽后的终止异常 | 开启 | 刚才请求频率受限并已中断，请继续未完成的任务 |
| HTTP 503 | 重试耗尽后的终止异常 | 关闭 | 刚才服务暂时不可用并已中断，请继续未完成的任务 |
| 其他 HTTP 错误 | 其他结构化终止型 4xx/5xx | 开启 | 刚才请求异常中断了，请继续未完成的任务 |
| 模型容量 | 已确认的模型容量终止异常 | 开启 | 刚才因模型容量限制中断了，请继续未完成的任务 |

“类型默认开启”不代表安装后立即发送：总开关默认关闭，用户必须主动开启总开关并完成系统授权。
活跃重试阶段只显示状态，不触发自动恢复。

## 配置模型与持久化

在 `ThreadBeaconCore` 增加稳定的异常分类枚举和配置模型：

- `AutoRecoveryIncidentType`：`http400`、`http429`、`http503`、`otherHTTP`、
  `modelCapacity`。
- `AutoRecoveryRule`：`isEnabled` 和 `prompt`。
- `AutoRecoverySettings`：总开关、各类型规则和版本字段。
- `AutoRecoverySettingsRepository`：使用 `UserDefaults` 编码保存，缺失字段回填默认值，损坏数据
  回退默认配置。
- `AutoRecoverySettingsStore`：向 SwiftUI 发布更改，并负责校验后持久化。

异常类型映射只依据已经结构化解析的 `ServiceIncidentKind`，不扫描会话正文，也不把未知错误文本
当作规则键。

## 数据流

1. `ThreadStatusStore` 在刷新后发现新的终止异常 episode。
2. 异常映射为稳定的 `AutoRecoveryIncidentType`；活跃重试不进入发送候选。
3. App 层读取总开关、对应规则和 Accessibility 授权状态。
4. 总开关或类型关闭时不发送、不产生误导性发送记录。
5. 规则启用但未授权时记录“未发送：需要 macOS Accessibility 授权”。
6. 已授权时把规则提示词传给 `SystemAccessibilityRecoverySender`；发送器不再使用固定提示词。
7. 发送器沿用任务 ID 导航、rename 标题校验、草稿保护、唯一发送按钮和 rollout 回读确认。
8. 结果写入自动恢复记录，同一任务的同一 episode 每次运行最多处理一次。

启动时已存在的历史异常继续只建立基线，不发送。用户修改规则后，不追溯处理已经观察过的异常。

## 失败处理

- 配置损坏：回退默认配置，总开关保持关闭。
- 提示词为空或超过限制：拒绝保存该字段，保留上一次有效值。
- 权限缺失或撤销：不调用 Accessibility 发送器，记录跳过原因。
- Codex 前台、存在草稿、目标身份不唯一或 AX 结构变化：失败关闭并记录具体阶段。
- 已触发发送但 rollout 未确认：记录“已触发但未确认”，不自动二次发送。
- HTTP 503 仍在重试：不触发；只有 `.failed` 阶段且用户开启 503 规则时才触发。

## 测试与验收

- 配置默认值、持久化、损坏数据迁移和提示词校验单测。
- 五种异常到规则分类的映射测试。
- 总开关、类型开关、授权状态和 `.retrying` / `.failed` 组合测试。
- 自定义提示词完整传入发送器和 rollout 确认解析器的测试。
- 同一 episode 去重、启动基线、503 默认关闭和无外部 CLI 回退测试。
- Debug 构建确认开发者诊断区存在；Release 构建确认相关文案和控件不进入界面。
- 简体中文、英文、浅色、深色和最小 Settings 窗口尺寸人工检查。

## 后续阶段

发送成功后恢复操作前的原前台 App，作为独立后续任务处理。实现时需要记录原前台应用、确保目标
发送已经完成或明确失败，再恢复焦点；同时避免在用户已主动切换应用时抢夺焦点。该能力不阻塞
本次自动恢复配置上线。
