# 自动恢复连续失败熔断设计

## 背景

ThreadBeacon 已能在检测到终止型 HTTP 异常、模型容量异常或连接重试耗尽后，通过 macOS
Accessibility 向对应 Codex 任务发送恢复提示词。现有链路会对同一个异常 episode 去重，但如果
任务在恢复后再次遇到同类异常，新 episode 仍会继续触发发送。

这意味着持续失败的任务可能形成“异常、自动恢复、再次异常、再次自动恢复”的循环。自动恢复已经
进入公开版本，因此需要在不削弱正常恢复能力的前提下，为每种异常增加可配置的连续失败熔断。

## 目标

- 按“任务 ID + 异常类型”限制连续自动恢复次数。
- 每种异常类型独立配置，默认最多自动恢复 `3` 次。
- 用户可以关闭某种异常的熔断限制，使其持续重试。
- 新的正常 `task_complete` 能可靠清零对应任务的连续计数。
- 熔断状态跨 App 重启保留，并提供按任务和异常类型手动解除入口。
- 自动恢复日志明确区分发送、失败、跳过和熔断。

## 非目标

- 不改变异常识别、终止状态判断和 episode 去重逻辑。
- 不引入按时间窗自动解封、指数退避或自动修改提示词。
- 不把 Accessibility 发送成功等同于任务正常完成。
- 不限制用户在 Codex App 中手工继续任务。
- 不读取会话正文，不保存任务标题、错误正文或提示词以外的会话内容。

## 方案比较

### 方案 A：持久化连续次数，正常完成后清零（采用）

按“任务 ID + 异常类型”保存连续自动恢复次数。真正开始 Accessibility 发送时次数加一；观察到
该任务更新的 `task_complete` 后，清除这个任务的所有异常类型计数。

该方案能表达“连续失败”的真实语义，并防止重启 App 绕过限制。代价是需要独立持久化状态，并把
任务完成证据传递给熔断状态存储。

### 方案 B：固定时间窗内限制次数

例如一小时内最多恢复三次。实现较简单，但任务已经正常完成后仍可能被时间窗内的旧失败误伤；时间
窗结束后，持续失败的任务又会自动恢复发送，因此不采用。

### 方案 C：仅在当前 App 生命周期计数

无需持久化，实现最轻量，但退出或重启 App 就会解除保护，不符合熔断的安全目标，因此不采用。

## 配置模型

`AutoRecoveryRule` 增加以下字段：

- `isCircuitBreakerEnabled`：是否启用连续失败熔断，默认 `true`。
- `maximumConsecutiveAttempts`：最大连续自动恢复次数，默认 `3`，有效范围 `1...20`。

每种 `AutoRecoveryIncidentType` 保存独立配置。关闭熔断时仍保留上一次次数值，再次开启后继续使用，
避免用户切换开关时丢失偏好。

配置模型版本递增。旧版本配置迁移时保留已有规则开关、提示词及提示词来源，并为新增字段补入“开启、
3 次”。损坏或越界的次数回退为 `3`，不影响其他合法字段。

## 熔断状态模型

新增独立的 `AutoRecoveryCircuitBreakerStore`，将状态持久化到 ThreadBeacon 的 Application
Support 目录。状态键由任务 ID 和异常类型组成，记录：

- 已真正开始的连续自动恢复次数。
- 最近一次计数对应的异常 episode ID。
- 最近一次开始恢复的时间。

状态文件不保存任务标题、异常正文或会话内容。写入使用原子替换；文件缺失或损坏时回退为空状态，
但不修改自动恢复总开关和用户规则。

## 计数与清零语义

1. 新终止异常通过现有 episode 去重后成为自动恢复候选。
2. 总开关或对应异常规则关闭时，不检查也不增加熔断次数。
3. 未获得 Accessibility 授权时记录“未发送”，不增加次数。
4. 对已经达到配置上限的“任务 ID + 异常类型”，拒绝发送并记录“已熔断”。
5. 通过安全检查并真正开始执行自动恢复时，次数加一并立即持久化。
6. 输入框冲突、目标校验失败、发送失败或发送后未确认，均不回滚已开始的次数。
7. Accessibility 操作锁被其他恢复占用、尚未真正开始发送时，不增加次数。
8. 观察到该任务时间晚于最近恢复尝试的正常 `task_complete` 后，清除该任务所有异常类型计数。
9. 用户在 Settings 手动解除时，只清除指定“任务 ID + 异常类型”的状态。

发送器确认恢复消息已经产生新的 `task_started`，只能说明恢复动作成功，不能清零。必须等到后续真实
`task_complete`，才能证明连续失败链路已经结束。

## 策略判断

策略判断保持失败关闭，顺序如下：

1. 检查自动恢复总开关和对应异常规则。
2. 检查 Accessibility 授权。
3. 如果该规则启用熔断，读取对应连续次数并与上限比较。
4. 未达到上限时返回发送决策；达到上限时返回结构化熔断决策。
5. 规则设置为“不熔断”时跳过次数上限判断，但仍记录次数，便于用户重新开启限制后得到保守保护。

更改配置后无需重写状态：降低上限可能使已有状态在下一次候选时立即熔断；提高上限或关闭熔断会在
下一次候选时恢复发送资格。

## Settings 交互

每种异常规则的展开区域增加：

- `连续失败熔断` 开关。
- 开启时显示 `Stepper`，范围为 `1...20`，文案为“最多自动恢复 N 次”。
- 关闭时显示“无限制，将持续尝试自动恢复”的说明。

在自动恢复记录附近增加紧凑的“当前熔断”区域，仅在存在熔断状态时显示。每行展示稳定的异常类型、
脱敏方式与现有日志一致的任务 ID、当前次数和上限，并提供“解除”按钮。解除只影响该任务和异常类型，
不修改规则配置，也不立即补发已经跳过的历史异常。

清空自动恢复日志只删除展示记录，不解除熔断，避免用户误把日志维护操作当成安全状态重置。

## 日志表现

`AutoRecoveryLogStatus` 增加 `circuitOpen`。达到上限后记录：

- 任务 ID。
- 异常类型和当前异常标签。
- 提示词快照。
- “连续自动恢复已达到 N 次，已停止发送”的本地化说明。

同一 episode 继续沿用现有去重，不重复追加熔断日志。手动解除后，也不追溯处理已跳过的 episode；
必须等待新的终止异常候选。

## 数据流

1. `ThreadStatusStore` 刷新任务快照并输出新的终止异常候选。
2. App 层读取对应规则、授权状态和 `AutoRecoveryCircuitBreakerStore` 当前计数。
3. 熔断策略返回关闭、待授权、已熔断或允许发送。
4. 允许发送时，在启动 Accessibility 操作后登记一次尝试，再执行现有安全发送链路。
5. 发送结果写入现有自动恢复日志；失败不自动重试。
6. 每次成功加载任务快照后，将主任务的 `completionEventAt` 交给熔断 Store。
7. Store 仅在完成时间晚于最近恢复尝试时间时清除该任务计数，并发布状态变化供 Settings 更新。

启动时的历史终止异常仍只建立基线，不发送。启动时读取到晚于持久化恢复尝试的 `task_complete` 可以
安全清零对应任务，避免 App 关闭期间已经正常完成却继续保留旧熔断。

## 失败处理

- 熔断状态文件损坏：回退为空状态，不阻断任务监听；数据源健康详情不暴露原始路径或内容。
- 状态持久化失败：本次运行继续保留内存计数，并在自动恢复日志记录稳定的本地错误说明。
- 配置次数越界：读取时归一化为默认 `3`。
- 任务完成时间早于或等于最近尝试时间：不清零，避免旧完成事件解除新熔断。
- 用户关闭熔断：允许后续候选继续发送，但 Accessibility 授权、草稿保护和单次执行锁仍然生效。
- 用户重新开启熔断：立即使用已有连续次数和新上限判断，不静默清零。

## 测试与验收

- 默认六种异常均启用熔断且上限为 `3`；HTTP 503 的自动恢复规则本身仍默认关闭。
- 旧版配置迁移后保留已有开关和提示词，并补齐熔断默认值。
- 次数 `1...20` 正常保存，缺失、零、负数和超上限值回退为 `3`。
- 同一任务同一异常的前三次允许发送，第四次返回熔断。
- 不同异常类型、不同任务之间计数互不影响。
- 未授权、规则关闭和操作锁占用时不增加次数。
- 发送开始后的失败或未确认仍保留已增加的次数。
- 新 `task_complete` 晚于最近尝试时清除该任务全部计数，旧完成事件不清除。
- App 重启后计数与熔断状态仍存在；状态文件损坏时安全回退。
- 关闭熔断后持续允许；重新开启时沿用已有次数。
- 降低和提高上限在下一次候选判断时生效。
- 手动解除只影响指定任务和异常类型，不补发历史 episode。
- 清空日志不会解除熔断。
- 简体中文、英文、浅色、深色及 Settings 最小宽度下完成 UI 检查。

## 发布边界

第一版不声称能修复上游服务异常。该能力只限制 ThreadBeacon 自动发送恢复消息的次数，并保证达到
上限后停止控制 Codex App。真实 400、429、503、其他 HTTP 终止错误、模型容量异常和连接中断样本
仍需持续观察，确认计数、正常完成清零和熔断日志符合预期。
